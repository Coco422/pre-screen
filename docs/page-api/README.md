# Pre-Screen 页面接口总览

这组文档按页面拆分，面向前端重构后的实际工作流整理。

- 接口入口统一以 Gateway 为准，也就是前端直接调用 `/api/admin/*` 和 `/api/public/*`。
- 文档同时标记三种状态：`已有`、`建议扩展`、`建议新增`。
- `建议扩展` 表示当前仓库已经有同名接口，但字段或筛选能力还不够。
- `建议新增` 表示当前仓库里还没有这个页面真正需要的接口。

## 候选人主状态建议

你描述的真实业务流程比当前 demo 状态更细，建议前后端统一成下面这一套：

| 业务阶段 | 建议状态 | 当前仓库常见状态 |
| --- | --- | --- |
| PDF 刚上传 | `已上传简历` | `解析中` |
| PyMuPDF + 多模态提取 | `信息提取中` | `解析中` |
| LLM 信息整理与核验 | `信息整理中` | `待审核` / `解析中` |
| 生成候选人专属题目 | `拟出卷中` | 无单独状态 |
| 题目草稿已保存 | `待发卷` | `待发卷` |
| 已生成考试链接和验证码 | `已发卷` | `待开考` |
| 候选人已进入考试 | `进行中考试` | `已开考` |
| 候选人已提交 | `已交卷` | `已完成` |
| HR 正在修分复核 | `评分复核中` | 无单独状态 |
| HR 确认通过筛选 | `已完成筛选` | 无单独状态 |
| 最终进入人才档案 | `已归档` | 无单独状态 |

## 当前代码和目标流程的主要差异

- 当前前端工作台通过 `tasks + candidates + results` 三个接口自己拼数据，建议改成单独的 `dashboard` 接口。
- 当前候选人列表接口只有 `task_id` 和 `status` 两个筛选条件，不够支撑岗位、待审核、已发卷、风险标记等真实筛选条件。
- 当前候选人详情还缺少证件照、原始提取结果、自我描述、比赛获奖经历、风险断言等字段。
- 当前结果详情只有查看能力，还没有 HR 修分、确认通过筛选、归档这几个关键接口。
- 当前候选人交卷后会直接进入 `已完成`，建议拆成 `已交卷 -> 评分复核中 -> 已完成筛选 -> 已归档`。
