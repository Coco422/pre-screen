# Pre-Screen UI 重构设计文档

## 1. 文档目标

本文档定义 Pre-Screen 前端 UI 重构的设计方向、信息架构、设计语言、组件约束和第一阶段实施范围。

本次重构不是局部美化，而是允许在项目早期阶段推翻现有前端页面结构、路由组织和视觉表达，重新建立一套适合本产品的企业级设计系统。

本文档默认服务于两个目标：

1. 让 HR 后台呈现出稳定、专业、可信赖的企业级操作台气质。
2. 让候选人考试端呈现出专注、清晰、低干扰的考试工作区气质。

## 2. 重构原则

### 2.1 视觉原则

- 现代化
- 科技感
- 严肃
- 可靠
- 蓝白主色

这里的科技感不是炫光、霓虹、复杂动画，而是通过有秩序的布局、精确的边框、克制的高亮、明确的状态和稳定的视觉层级来建立。

### 2.2 产品表达原则

前端页面必须减少无意义的解释性文案，优先展示：

- 图像与图标
- 关键 UI 结构
- 状态
- 数据
- 操作入口
- 必要提示

禁止继续沿用以下表达方式：

- 工作流脑内独白式文案
- 系统正在努力帮你做事的叙述
- 冗长的产品说明
- 情绪化、口语化、营销化文案

### 2.3 实施原则

- 允许推翻当前 UI 实现
- 允许重组前端页面结构与路由
- 第一优先级是高保真贴近确认后的设计稿
- 不以“工程化优化”为理由削弱用户明确认可的视觉内容
- 优先复用成熟组件，避免重复造轮子

## 3. 组件与技术约束

### 3.1 Element Plus 使用原则

本项目 Web 端在重构过程中应优先使用 [Element Plus](https://element-plus.org/) 的成熟组件能力，而不是手写重复的基础交互控件。

优先复用的组件类别包括：

- 表单输入
- Select / Cascader / Date Picker
- Tabs
- Table
- Pagination
- Dialog / Drawer
- Dropdown
- Tooltip
- Tag
- Badge
- Progress
- Skeleton
- Empty
- Message / Notification

允许自定义视觉外观，但应尽量建立在 Element Plus 的组件能力之上，而不是重新实现行为逻辑。

### 3.2 可自定义层

以下内容可以围绕 Element Plus 做视觉重构和布局封装：

- 页面壳子
- 左侧导航
- 顶部全局栏
- 指标卡
- 业务状态标签规范
- 页面区块容器
- 登录页品牌视觉
- 考试端工作区结构

### 3.3 不应重复造轮子的范围

除非出现明确的交互限制，不应自行重复实现：

- 通用输入框行为
- 常规下拉菜单
- 常规分页器
- 常规表格交互
- 消息提醒基础能力
- 通用弹窗与抽屉基础能力

## 4. 产品信息架构

### 4.1 两大产品区

整个前端拆分为两个产品区：

#### 1. Admin Console

面向 HR / 招聘运营。

核心目标：

- 查看状态
- 做出筛选判断
- 推进发卷流程
- 查看结果与风险信息

#### 2. Exam Workspace

面向候选人。

核心目标：

- 快速进入考试
- 专注作答
- 稳定保存与提交

### 4.2 路由骨架

Admin 路由：

- `/login`
- `/admin/dashboard`
- `/admin/tasks`
- `/admin/tasks/new`
- `/admin/tasks/:taskId`
- `/admin/candidates`
- `/admin/candidates/:candidateId`
- `/admin/candidates/:candidateId/edit`
- `/admin/papers/:paperId`
- `/admin/results`
- `/admin/results/:resultId`

Exam 路由：

- `/exam/:token/start`
- `/exam/:token/session`
- `/exam/:token/submitted`

说明：

- 登录页保持独立，不挂在后台壳子里。
- Admin 页面统一挂载在后台壳子下。
- Exam 端按“验证 -> 作答 -> 提交完成”拆成 3 段独立页面。

### 4.3 后台导航层级

后台左侧导航先按完整产品结构建立，即使短期内部分页面仍为占位页，也应先保留入口层级：

- 工作台
- 任务中心
- 候选人
- 考卷管理
- 结果中心
- 风险管理
- 系统设置

未在第一阶段实现的页面允许先使用标准占位页承接，后续逐步替换。

## 5. 设计语言规范

### 5.1 色彩系统

主色系统固定为蓝白体系：

- 品牌蓝：主操作、激活态、关键高亮
- 浅天蓝：背景层、选中区、轻提示块
- 白色：主界面表面
- 深灰蓝：主文字
- 中灰：次级文字、分割线、辅助信息

状态色仅用于语义表达：

- 成功
- 警告
- 风险
- 信息

限制：

- 全站只有一个主强调色，即蓝色
- 不允许其他高饱和颜色与主色争夺视觉中心
- 后台页面避免大面积高饱和渐变

### 5.2 背景与材质

- 登录页允许使用较强的科技感背景和主视觉插图
- 后台内页背景应更克制，以浅蓝灰和白色为主
- 可使用极弱的网格、线条、光斑营造科技感，但不能影响信息可读性

### 5.3 字体与层级

文字层级控制在以下范围：

- 页面主标题
- 区块标题
- 表头 / 字段标签
- 正文 / 标签文字
- 辅助信息

要求：

- 标题短
- 标签准
- 提示克制
- 避免长段说明

### 5.4 圆角、边框、阴影

- 圆角中等偏大，但不能软萌
- 边框细且浅
- 阴影轻量、干净、少层级
- 靠结构、留白和秩序感建立高级感

### 5.5 动效

只保留克制的必要动效：

- 页面进入淡入
- 轻微位移过渡
- Hover / Focus 反馈
- 加载与切换的短过渡

避免：

- 大范围浮动动画
- 高频闪动
- 干扰操作判断的装饰动效

## 6. 后台统一页面骨架

所有 Admin 页面统一遵守以下骨架：

- 左侧固定导航栏
- 顶部全局信息栏
- 页面主区域
- 页内操作区
- 列表 / 表单 / 详情主体区

顶部全局信息栏只保留：

- 当前页面标题
- 少量系统级提醒图标
- 当前登录用户
- 可选主操作按钮

不再展示大段“页面说明式文案”。

## 7. 候选人考试端骨架

考试端采用三段式结构：

### 7.1 考试验证页

显示内容：

- 考试名称
- 岗位名称
- 时长
- 验证码输入
- 开始考试按钮

### 7.2 在线作答页

结构建议：

- 顶部状态栏
- 左侧题目导航
- 中间答题区
- 右侧运行结果区或辅助区

### 7.3 提交成功页

显示内容：

- 明确的提交成功状态
- 提交时间
- 退出按钮

## 8. 第一阶段实施范围

第一阶段不追求一次重做全部页面，而是先完成 3 个样板页和全局设计底座：

1. 登录页
2. HR 工作台
3. 候选人列表页

这 3 页将作为后续整站迁移的母版。

## 9. 第一阶段样板页要求

### 9.1 登录页

目标：

- 建立品牌气质
- 建立蓝白科技感首屏
- 建立表单输入、按钮和登录卡片规范

页面结构：

- 左侧品牌区与大型主视觉
- 右侧登录卡片

只保留 `HR 登录`，不保留候选人登录视觉。

允许为登录页单独生成一张符合风格的主视觉插图。

必须删除当前实现中的以下内容：

- 工作流步骤说明
- 大段登录后行为解释
- 过强的过程性反馈文案

### 9.2 HR 工作台

目标：

- 建立后台统一壳子
- 建立指标卡、导航和主体布局规范
- 建立“数据优先、状态优先”的后台表达

页面内容应以以下模块为主：

- 顶部指标卡
- 待处理候选人
- 最近任务
- 最新结果 / 风险提醒
- 快捷入口

必须删除当前实现中的以下内容：

- 解释型 Hero 卡
- “建议下一步”大块叙事
- 工作流说明式文案

### 9.3 候选人列表页

目标：

- 建立整站表格系统
- 建立筛选器系统
- 建立状态标签和操作列规范

页面结构：

- 页面标题区
- 顶部筛选条
- 主表格
- 分页区

列表页必须从当前的卡片流改为表格驱动页面。

必须删除当前实现中的以下内容：

- 候选人卡片宫格
- 大段摘要展示
- 技能标签大面积铺开
- 页面级大号“继续处理”按钮

## 10. 后续页面迁移顺序

第一阶段完成后，建议按以下顺序迁移剩余页面：

1. 任务中心
2. 候选人详情页
3. 编辑画像页
4. 考卷编辑与发布页
5. 结果中心
6. 结果详情页
7. 考试验证页
8. 在线作答页
9. 提交成功页

## 11. 后续页面迁移规则

### 11.1 列表型页面

优先使用：

- Element Plus Table
- 筛选器
- 标签
- 分页

不再使用信息密度低的卡片流表达高频业务列表。

### 11.2 详情型页面

优先使用双栏或三栏布局，强调：

- 基础信息
- 结构化信息
- 操作区
- 风险区

### 11.3 编辑型页面

优先采用分组表单与稳定的保存反馈。

### 11.4 工作区页面

如考试端与代码题区，应优先保障：

- 专注
- 稳定
- 低干扰
- 明确的状态反馈

## 12. 第一阶段完成标准

第一阶段完成后，应满足以下结果：

1. 整站新气质已经建立。
2. 后台新壳子已经建立。
3. 全局设计语言已经真实落地，而不是停留在口头描述。
4. 登录页、工作台、候选人列表页可作为后续页面的设计母版。
5. 原有冗长、解释型、低效的信息表达被整体淘汰。

## 13. 实施备注

- 本次 UI 重构允许改变现有前端组织方式。
- 为避免重复造轮子，基础交互能力应优先复用 Element Plus。
- 高保真贴近已确认设计稿是默认前提。
- 若设计稿与现有页面逻辑存在冲突，优先保留设计稿中的信息结构与视觉原则，再对实现做适配。
